## 4. Parameterized Context Tools: Security & Negative Tests

We executed a dedicated smoke test script (`security_smoke_test.py`) to verify tool boundaries, input parameterization, limits enforcement, and SQL injection safety.

### 4.1 Safe Tool Test Results
*   **`get_player_context_packet`**: Requesting `"Patrick Mahomes"` with scoring profile `"ppr"` works correctly and returns a clean packet or an explicit unavailable message. No SQL parameters are accepted.
*   **`search_players`**: Searching for `"Jefferson"` with a requested limit of `5` returns a bounded result set. The limit is enforced correctly.
*   **`get_rankings_slice`**: Fetching positional rankings for `"WR"` in season `2026` constructs a query targeting `analytics_pigskin_rankings` using parameters.
*   **`get_fraud_watch_candidates`**: Returns Candidates from `analytics_fraud_watch`.
*   **`get_trade_player_history`**: Correctly delegates to the trade history utility.
*   **`compare_players`**: Evaluates comparisons for Chase and Jefferson. Rejects evaluation if the list exceeds the maximum comparison limit of 6.
*   **`get_context_event_leads`**: Returns events and search leads. Stored search leads are explicitly labeled as leads, not verified truth.

### 4.2 Malicious Input Test Results
- All malicious search strings, enums, format IDs, and limits injection payloads were safely parameterized and clamped.
- No raw/source table query executed, and no SQL injection was possible.
- No database errors or stack traces leaked. (Pass)
